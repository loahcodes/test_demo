A ideia do GIT é criarmos um repositório master onde ficarão as implementações em produção. Enquanto isso, podemos desenvolver um espécie de ambiente de testes para aprimoramentos, debugs e outras features desenvolvidas em branches secundárias. Apenas quando concluídas e prontas para entrar em produção mesclamos com a master.

- git config --help
abre no navegador a documentação do git

- git --version
checar que versão do git temos instalada

- pwd
mostra onde estou agora, o caminho.

- mkdir <nome pasta>
criar pasta de repositório
por padrão vai ser criada logo em user, pois é onde estamos ao abrir o GIT bach

- rmdir <nome pasta>
remove pasta

- cd <caminho>
desloca pelos diretórios do pc

- cd ..
volta um estágio no caminho

- git init
cria a branch master no local onde estamos

- git status
resume o status dos itens na pasta do nosso repositório

- ctrl + l
limpa a tela


IDENTIFICAÇÃO
- git config --global user.email "email"
identifica seu email para os logs
- git config --global user.name "nome"
identifica seu user name para os logs

Acredito que para usar no github, tem que ser o mesmo username que tem em (signed in as <nome>, no meu caso é loahcodes)


- git config --list
checar as configurações atuais

- git add <nome arquivo>
passa a monitorar um arquivo na pasta do repositório

- git add .
adiciona todos os arquivos

- git commit -m "texto da msg resumo do log"
commita uma mudança, aplica ela, permitindo criar uma mensagem com um breve resumo sobre o que foi alterado, tornando mais fácil acompanhar as mudanças.

- ls
mostra todos os arquivos no branch/pasta

- git remote add origin <caminho repositório online>
para linkar com um repositório remoto, o github por exemplo:
git remote add origin https://github.com/loahcodes/test_demo.git

- git remote -v
checar associação com repositorio remoto

- git push origin master
envia arquivos para o origin, do master
obs: mesmo tendo enviado os arquivos, as vezes o cash impede o navegador de perceber, basta ir em more tools > clear browsing data

- git diff
mostra a diferença nos arquivos entre o que tem commitado no repositório (ou pelo menos adicionado para commit) e o arquivo no estado atual.

- git branch <nome branch>
cria um novo branch

- git branch -v
para ver minhas branchs, a verde é a que estou agora

- git checkout <nome branch>
vai para o branch que você chamou

TRABALHANDO COM BRANCHES
- criar a branch para poder trabalhar nela sem mexer no original
- fazer toda a sua atividade
- mesclar com a original
- deletar a branch
- enviar a alteração para o servidor (push)
*senão ele não vai saber que a branch foi deletada

- git merge <nome branch>
usar o comando no local onde você quer seja mesclado e chamar no comando a branch que você quer trazer.
ex:	se quer trazer a first_branch pra master
	1o. - git checkout master
	2o. - git merge first_branch
assim, a master irá receber tudo o que há na first_branch.

- git branch -D <nome branch>
deleta a branch, ter cuidado para não estar na branch que você quer deletar

- touch <nome arquivo>
cria arquivo nas pasta onde estamos

- notepad <nome arquivo>
abre arquivo notepad

- git log
mostra o log de alterações

- git log --pretty=online
mostra os logs resumidos em 1 linha cada

- git clone <url do repositório remoto>
clona um repositório para outro

- git reset --soft <hash do commit>
desfaz os commits posteriores ao hash que indiquei.
os arquivos continuam tendo sido mudados e adicionados, porém, não commitados.
obs: fazer com parcimônia e de preferência apenas quando só você estava mexendo nos arquivos.

- git revert <hash do commit>
desfaz o commit do hash, apenas ele, e cria um commit indicando que o commit foi revertido.
